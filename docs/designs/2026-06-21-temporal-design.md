# DevFocus + LLMCompare 时间变化设计

> **目标**：两个站都要表达"内容随时间变化"，让用户感受到信息是活的、在流动的。

---

## 一、DevFocus — 趋势热力图

### 核心理念
用 **话题聚类 × 时间** 的热力图，展示哪些开发者话题在升温、哪些在降温。

### 数据基础
- `data/5-history/` 已有 8 天快照（6/13-6/20），每天 ~300 条
- 7 个数据源：HN, GitHub, PH, 掘金, 知乎, 36氪, InfoQ
- 每条有 title, score, source, time, tags

### 数据管线改动

**新增脚本**：`data/3-process/build_trends.py`

1. **话题聚类**：对历史快照的标题做关键词提取（TF-IDF 或简单 n-gram）
   - 中文分词：jieba（已有依赖或用标准库 re 按标点/空格切）
   - 英文：按停用词过滤 + bigram
   - 聚类：同一关键词/短语出现 ≥2 次 → 归为一个话题
   
2. **热度计算**：每个话题每天的热度 = 命中条目的 score 之和

3. **输出**：`4-final/trends.json`
   ```json
   {
     "generated_at": "...",
     "period": "2026-06-13 ~ 2026-06-20",
     "topics": [
       {
         "keyword": "大模型",
         "count": 12,
         "trend": "rising",  // rising | falling | stable | new
         "heat_by_date": {
           "2026-06-13": 3,
           "2026-06-14": 5,
           "2026-06-15": 8,
           ...
         },
         "sample_titles": ["智谱创始人...", "DeepSeek V4..."],
         "sources": ["36kr", "zhihu", "infoq"]
       }
     ],
     "source_activity": {
       "36kr": { "2026-06-13": 18, "2026-06-14": 20, ... },
       "infoq": { ... },
       ...
     }
   }
   ```

4. **趋势判断**：
   - `new`：最近 2 天首次出现
   - `rising`：近 3 天热度均值 > 前 3 天均值 × 1.5
   - `falling`：反之
   - `stable`：其他

## 二、LLMCompare — 每日变化卡片

### 核心理念
用 **事件流** 展示模型排名的动态变化，让用户一眼看到"今天发生了什么"。

### 数据基础
- 目前无历史数据，需新增快照机制
- 每天 GitHub Actions 刷新 ranking.json（~60 模型）

### 数据管线改动

**1. 新增快照保存**（pipeline.py Phase 3 之后）：
```python
# 保存当日快照
HISTORY_DIR = DATA / "5-history"
HISTORY_DIR.mkdir(exist_ok=True)
shutil.copy(DATA / "4-final" / "ranking.json", HISTORY_DIR / f"{TODAY}.json")
```

**2. 新增脚本**：`data/3-process/build_changes.py`

对比今天和昨天的 ranking.json，提取：
- **新上榜**：今天有、昨天没有的模型
- **下榜**：昨天有、今天没有的模型
- **排名变动**：综合分排名变化 ≥3 位的模型
- **价格变化**：price_input 或 price_output 变化 > 10%
- **智商变化**：intelligence_index 变化 > 5
- **速度变化**：median_tps 变化 > 20%

输出 `4-final/changes.json`：
```json
{
  "generated_at": "...",
  "date": "2026-06-21",
  "compare_with": "2026-06-20",
  "summary": {
    "new_models": 1,
    "dropped_models": 0,
    "price_changes": 3,
    "ranking_changes": 5
  },
  "changes": [
    {
      "type": "new",
      "model": "Qwen3.7-72B",
      "detail": "智商 62, 速度 85 TPS, $0.3/M tokens",
      "icon": "🆕"
    },
    {
      "type": "price_drop",
      "model": "Claude Sonnet 4",
      "field": "price_input",
      "old": 3.0,
      "new": 1.5,
      "change_pct": -50,
      "icon": "💰"
    },
    {
      "type": "ranking_up",
      "model": "DeepSeek V4",
      "old_rank": 8,
      "new_rank": 3,
      "change": 5,
      "icon": "🔺"
    }
  ]
}
```

**3. 历史数据回填**（一次性）：
从 GitHub Actions 历史 PR 中提取旧 ranking.json，填充 `5-history/`。

## 三、实施顺序

| 阶段 | 站点 | 内容 | 工作量 |
|------|------|------|--------|
| Phase 1 | LLMCompare | 快照机制 + build_changes.py + 变化卡片 | 2h |
| Phase 2 | DevFocus | build_trends.py + 热力图页面 | 3h |
| Phase 3 | LLMCompare | 模型详情页趋势线 | 1h |
| Phase 4 | DevFocus | 源活跃度图 + 移动端适配 | 1h |

**先做 LLMCompare**，因为数据管线改动小（只需加快照），且变化卡片信息密度高、开发快。
